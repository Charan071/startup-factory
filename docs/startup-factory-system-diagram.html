Startup Factory system flow with runtime wiring, prompt choreography, and the planned next graph.

This page turns the architecture notes into a browser-friendly flow. It shows what is already implemented in the current startup_factory package, what the typed contracts look like, where OpenAI and LangSmith sit in the runtime, and where optional deep-agent workers can slot in later without taking over the whole application.

Entry Layer
Current entrypoint

CLI request

startup_factory.cli parses --brief, --top-k, and repeatable constraints before creating a validated run request.

  • python -m startup_factory.cli --brief "..."
  • Optional `--json` output for machine-readable runs
  • Small surface area, easy to debug locally
Future interface

API and UI

The architecture doc reserves a service boundary for FastAPI and a future browser UI without changing the graph core.

  • FastAPI router can call the same graph function
  • Future UI can submit briefs and render reports
  • No orchestration logic needs to move out of LangGraph
Validation

RunRequest contract

The first hard gate is RunRequest: non-blank brief, optional constraints, and a safe top_k range.

  • Blank briefs are rejected
  • `top_k` is bounded from 1 to 10
  • Constraint parsing stays explicit and debuggable
Output goal

Ranked report

The entry layer is thin because the real product value is a final ranked startup report with clear fields and scores.

  • Problem
  • Solution
  • ICP, MVP, GTM
  • Critic summary and score breakdown

Node Execution Matrix
Node 01
industry_scan
Live
First model call in the pipeline. It translates the brief into a short list of real operating environments with visible inefficiency.
Consumes
user_brief, constraints
Prompt and output
prompts/industry_scan.txt to IndustryScanResult
Prompt contract
  • Return 4 to 6 industries.
  • Focus on hidden operational inefficiencies, not AI hype.
  • Stay tightly anchored to the user brief and its domain.
Node 02
problem_mine
Live
Narrows industry observations into recurring workflow failures with actual buyers, so the system does not jump from vague markets to product ideas.
Consumes
user_brief, constraints, industries
Prompt and output
prompts/problem_mine.txt to ProblemMiningResult
Prompt contract
  • Return 4 to 6 problems.
  • Prefer narrow recurring workflow failures over generic inefficiency.
  • Favor painful B2B problems with clear budget owners.
Node 03
generate_ideas
Live
Converts validated pain into product candidates with enough commercial structure to be ranked instead of remaining raw brainstorm fragments.
Consumes
user_brief, constraints, problems
Prompt and output
prompts/generate_ideas.txt to IdeaGenerationResult
Prompt contract
  • Produce 4 to 6 ideas total.
  • Favor recurring-revenue B2B software feasible for a small team.
  • Avoid broad platforms, generic marketplaces, and all-in-one products.
Node 04
critique_ideas
Live
Acts as the first adversarial checkpoint so the graph can punish weak assumptions before scoring makes a bad idea look falsely precise.
Consumes
user_brief, ideas
Prompt and output
prompts/critique_ideas.txt to CritiqueResult
Prompt contract
  • Critique every idea provided.
  • Attack urgency, willingness to pay, and adoption assumptions.
  • Keep the critic summary concise but actionable.
Node 05
score_ideas
Live
Transforms drafts plus critiques into comparable scores, then enforces ranking and truncation rules inside the graph rather than inside the prompt.
Consumes
user_brief, ideas, critiques, top_k
Prompt and output
prompts/score_ideas.txt to ScoreResult
Prompt contract
  • Score every idea provided.
  • Penalize weak moats, low urgency, or obvious commoditization.
  • Favor urgent, buyer-funded, small-team-feasible opportunities.
Node 06
finalize_output
Serializer
This step is intentionally non-LLM. It packages the ranked output into a stable report contract that the CLI and future API can both trust.
Consumes
user_brief, ranked_ideas
Output
FinalReport stored under final_output
Why it matters
  • Keeps report formatting separate from model prompting.
  • Makes markdown and JSON output routes share one final object.
  • Provides a clean boundary for tests and future API responses.

Contracts and State
Graph state

WorkflowState

The graph passes a typed state object through every node so intermediate artifacts stay inspectable instead of disappearing inside prompts.

  • user_brief, constraints, top_k
  • industries, problems, ideas, critiques
  • scores, ranked_ideas, final_output
Pydantic models

Strict schemas

Every node targets an explicit result model, which is the main protection against malformed model JSON polluting the graph.

  • IndustryScanResult
  • ProblemMiningResult
  • IdeaGenerationResult
  • CritiqueResult
  • ScoreResult
  • FinalReport
Final object

Serializable output

The final report is shaped for both humans and code: markdown for quick reading, JSON for storage or downstream tooling.

  • Generated timestamp
  • Original brief
  • Top ideas list
  • Score breakdown per idea
RunRequest --> graph.invoke(state) state["industries"] --> state["problems"] --> state["ideas"] state["ideas"] + state["critiques"] --> state["scores"] --> state["ranked_ideas"] state["ranked_ideas"] --> FinalReport

Runtime and Observability
Model runtime

OpenAI structured LLM

OpenAIStructuredLLM asks for a JSON object, parses the response, and validates it into the exact Pydantic model required by the node.

  • Default model: gpt-4o-mini
  • Empty responses fail fast
  • Invalid JSON becomes a runtime error immediately
Prompt system

Prompt-per-node

Each node loads its own prompt file, which keeps orchestration code small and lets prompt changes happen without rewriting the graph.

  • prompts/industry_scan.txt
  • prompts/problem_mine.txt
  • prompts/generate_ideas.txt
  • prompts/critique_ideas.txt
  • prompts/score_ideas.txt
Tracing

LangSmith visibility

The OpenAI client is wrapped for tracing, so once LangSmith env vars are configured you can inspect prompt, response, timing, and failures per node.

  • LANGSMITH_TRACING
  • LANGSMITH_API_KEY
  • LANGSMITH_PROJECT
  • Useful for real run evaluation, not just tests
Structured call chain
This is the trust boundary that makes the graph debuggable: prompt text in, JSON out, strict model validation, then typed state updates.
Input

Prompt assembly

Node prompt + serialized state subset are assembled into the exact system and user messages sent to the model.

→
Validation

Strict parse and merge

JSON is parsed, validated into a result model, then only the expected fields are merged back into WorkflowState.

Runtime choice

Validation lives in code

The prompts ask for structure, but the real enforcement happens after the response lands. That keeps failure modes visible and testable.

Flow choice

Ranking lives in the graph

Sorting, truncation, and missing-ID checks happen in Python, which keeps final ranking logic deterministic instead of prompt-dependent.

Ops choice

Tracing stays optional

LangSmith is ready for deeper inspection, but local runs still work as a small CLI workflow without forcing extra infrastructure.

Future Specialist Workers
Deep agents later

Research-heavy industry scan

If industry discovery becomes tool-heavy, this node can delegate outward research without forcing the entire workflow to become one large autonomous agent.

Validation lane

Competitor and market checks

A future validation worker can pull competitor evidence, reviews, job posts, or adjacent signals to sharpen scoring and filter out fake opportunities.

Red team lane

Stronger critique engine

The critique stage is a good place for adversarial specialists because the output is bounded: stress the idea, list risks, and hand the result back to the graph.

LangGraph remains the orchestrator
Specialists stay inside selected nodes
No autonomous infinite loop in V1
